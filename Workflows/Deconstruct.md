# Deconstruct Workflow

Use this workflow to break down any problem, system, process, cost, or decision into its fundamental parts and irreducible facts.

## When to use

Use this workflow when:

- the problem feels vague or overly complex
- cost or effort seems "just how it is"
- a workflow feels heavy but the cause is unclear
- a system needs to be understood before redesign
- a user question is trapped inside a bad frame
- you need to expose what the problem is actually made of

This is usually the first step in a full first-principles analysis.

## Core question

What is this really made of?

Do not accept the packaged description. Break the subject down into actual components, steps, mechanics, costs, dependencies, incentives, and behaviors.

## Process

### Step 1: Name the subject clearly

Write:

- what is being analyzed
- what outcome matters
- what people usually say it is

Example:
"We are deconstructing tutorial production" is better than "we are deconstructing content."

### Step 2: List the stated components

List how the subject is commonly described.

Ask:

- What parts do people usually say this includes?
- What does the industry description contain?
- What is treated as obviously part of the system?

Do not judge yet. Capture the default picture first.

### Step 3: Break each component into actual constituents

For each stated component, ask:

- What is this actually made of?
- What are the real steps?
- What are the actual inputs?
- What are the hidden dependencies?
- What does it cost in time, money, attention, coordination, or trust?
- What is the minimum viable version of this component?

Go from labels to mechanics.

### Step 4: Identify fundamental facts

Keep only what remains when you strip away convention.

Examples of fundamental facts:

- user behavior patterns
- mathematical relationships
- legal requirements
- physical limits
- true dependencies
- actual cost drivers
- actual time bottlenecks
- empirically observed constraints

Do not treat market slogans, historical habits, or standard pricing as fundamental facts.

### Step 5: Find the gap between stated and actual

Compare:

- what people think the system is
- what the system is actually composed of

Look for:

- inflated categories
- bundled complexity
- unnecessary intermediaries
- mislabeled costs
- misunderstood bottlenecks
- hidden repeated work

## Output format

Use this structure when helpful:

### Subject
What is being deconstructed?

### Default framing
How is the problem usually described?

### Stated components
List the default parts.

### Actual constituents
For each part, show the underlying mechanics, inputs, and real costs.

### Fundamental facts
List only the facts that survive decomposition.

### Key gaps
Show where the default framing differs from reality.

### Implications
State what becomes possible now that the structure is clearer.

## Quality bar

A good deconstruction should make the problem feel:

- smaller
- clearer
- less mystical
- more measurable
- easier to redesign

If the output only renames the obvious parts, go deeper.

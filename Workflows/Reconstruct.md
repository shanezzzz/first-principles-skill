# Reconstruct Workflow

Use this workflow to build a cleaner solution from the remaining hard constraints and verified facts.

## When to use

Use this workflow when:

- Deconstruct and Challenge have clarified the problem
- the current solution is clearly overbuilt
- the user wants a better design from scratch
- multiple options need to be evaluated from fundamentals
- the answer should end in a real recommendation

This usually follows Deconstruct and Challenge.

## Core question

If only the real constraints remained, what would we build?

Optimize for function, not inherited form.

## Process

### Step 1: State only the hard constraints and verified facts

Do not carry forward:

- soft constraints
- default patterns
- competitor mimicry
- unvalidated assumptions
- process baggage

Write only what must remain true.

### Step 2: Define the function

Describe the real outcome in plain terms.

Examples:

- not "build an app"
- but "let users accomplish X in a low-friction way"

- not "support a complex content pipeline"
- but "help the audience reach understanding quickly and consistently"

- not "use premium pricing tiers"
- but "capture value in proportion to the result created"

### Step 3: Generate clean designs

Create two or three plausible approaches if needed.

Prefer designs that optimize one or more of:

- simplicity
- speed of learning
- cost efficiency
- user value
- reduced coordination
- reduced surface area
- clearer ownership
- faster time-to-value

Do not default back to the conventional design unless it wins on fundamentals.

### Step 4: Compare options against the real function

For each option, ask:

- Does it satisfy the hard constraints?
- Does it achieve the real goal?
- Is it simpler than the inherited solution?
- Is it cheaper?
- Is it faster to test?
- Does it reduce dependence on fake constraints?
- What new risk does it introduce?

### Step 5: Recommend the best current path

Choose the option that best satisfies the real function with the least unnecessary structure.

Do not end with "it depends" unless the uncertainty is truly unresolved. If uncertainty remains, state what fact decides the choice.

### Step 6: End with implementation or experiment

Give one of:

- a minimal experiment
- a first implementation sequence
- a decision rule
- a priority order

The user should be able to act immediately.

## Output format

Use this structure when helpful:

### Hard constraints and verified facts
List only what is real.

### Function to optimize
State the real outcome.

### Candidate designs
Show one or more approaches.

### Comparison
Compare them on simplicity, cost, speed, value, and risk.

### Recommended solution
State the best current answer and why.

### What gets removed
Show what complexity, cost, or structure disappears.

### Next move
State the first action, test, or implementation step.

## Quality bar

A good reconstruction should feel:

- simpler than expected
- grounded in reality
- easier to test
- less dependent on convention
- more focused on the actual outcome

If the final answer still looks like the old solution with minor edits, you probably reconstructed too weakly.

# Prompt Transforms

Use this file to convert ordinary user questions into first-principles analysis frames.

## Core rule

Translate the user's surface question into:

1. desired outcome
2. assumed method
3. hidden assumptions
4. likely constraints
5. the smallest useful redesign question

Do not stay trapped inside the user's initial framing if it is only one possible method.

## Examples

### Surface question
How do I make this look more professional?

### Transform
- What effect does "professional" need to create: trust, conversion, authority, clarity, or price justification?
- Which visible elements actually create that effect?
- Which polishing tasks are inherited aesthetics rather than value drivers?

---

### Surface question
Should I build an app?

### Transform
- What user outcome needs to be delivered?
- Is an app the necessary vehicle or just one possible format?
- What is the smallest form that could validate adoption first?

---

### Surface question
Should I add this feature?

### Transform
- What user outcome is currently blocked?
- Is the feature the smallest way to unblock it?
- Is the feature solving the core problem or compensating for another weakness?

---

### Surface question
Why is this workflow so slow?

### Transform
- Which step is the actual bottleneck?
- Which steps create value and which exist for coordination, reassurance, or legacy reasons?
- If rebuilt today, which steps would remain?

---

### Surface question
How do I grow faster?

### Transform
- Is the bottleneck attention, message relevance, trust, conversion, product value, retention, or repeatability?
- What part of the system is truly limiting progress?
- Which effort is producing noise instead of leverage?

---

### Surface question
How do I make better tutorials?

### Transform
- Better in what dimension: clarity, retention, authority, conversion, or production efficiency?
- What does the viewer need by the end?
- Which parts of the workflow improve that outcome and which parts just consume time?

## Fallback transform

When the user's question is vague, convert it into:

- What is the real outcome?
- What method is being assumed?
- What constraints are definitely real?
- What constraints might be inherited?
- If we started from zero, what is the simplest version that still works?

# Operations, Workflow, and Process Design

Use this file for SOPs, automation, internal workflow, handoffs, approvals, team coordination, tool chains, and efficiency questions.

## Core orientation

Do not assume the existing process is necessary.

Start from:

- the final operational outcome
- the risk the process is trying to control
- the information that must move
- the approvals that truly matter
- the minimum steps required

## Decompose any workflow into:

- trigger
- inputs
- transformations
- decisions
- approvals
- handoffs
- outputs
- failure modes
- monitoring needs

Then ask which parts are essential.

## Common operational illusions

Watch for:

- extra handoffs caused by org boundaries rather than task logic
- approvals added after old incidents and never revisited
- duplicated data entry across tools
- reporting work that does not affect decisions
- meetings compensating for poor information flow
- tool fragmentation treated as unavoidable

## Useful output structure

- Desired outcome
- Assumptions
- Basic workflow facts
- Hard constraints
- Soft constraints
- Rebuilt process
- First low-risk simplification
- Metrics to watch

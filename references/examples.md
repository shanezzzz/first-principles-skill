# Examples

Use these examples to recognize the structure of first-principles reasoning.

Pattern:

- Surface problem
- Assumptions
- Basic facts
- Constraint split
- Rebuilt design
- Minimal test

## Example 1: Tutorial production feels too slow

### Surface problem
Creating tutorials takes too long.

### Assumptions
- High-quality tutorials require heavy editing
- Long-form is necessary
- Every tutorial must be produced from scratch
- Higher polish always creates better audience outcomes

### Basic facts
- Audience value comes mainly from clarity, proof, pacing, and usefulness
- Many editing tasks improve polish more than understanding
- Reusable systems reduce repeated thinking cost
- One strong source artifact can produce many outputs

### Constraint split

Hard constraints:
- The material must be correct
- The tutorial must be understandable
- The content must hold enough attention to deliver value

Soft constraints:
- Cinematic editing
- Unique production workflow for every tutorial
- Separate production logic for every platform
- Large amounts of manual polishing before publishing

### Rebuilt design
Use a repeatable tutorial pipeline:

idea -> proof/demo -> structured script -> core recording -> derivative cuts -> templated distribution

### Minimal test
Ship one tutorial with the reduced pipeline and compare:
- production hours
- completion rate
- saves or shares
- audience response

## Example 2: AI product MVP feels too small to launch

### Surface problem
The product does not feel complete enough to ship.

### Assumptions
- Users expect a broad feature set
- A narrow tool looks weak
- Missing secondary features will kill adoption

### Basic facts
- Users adopt products that solve one painful job clearly
- Early adoption often depends more on outcome than completeness
- Extra features can dilute the main value proposition
- Shipping early increases learning speed

### Constraint split

Hard constraints:
- The core promised outcome must work
- The product must be usable enough to test real demand

Soft constraints:
- Full dashboard
- Team features
- Settings depth
- Broad competitor parity

### Rebuilt design
Ship the smallest product that solves one painful outcome well.

### Minimal test
Launch with one clear promise to a narrow audience and measure:
- activation
- repeat usage
- willingness to pay
- feedback on missing features

## Example 3: CAC feels too expensive

### Surface problem
Customer acquisition cost is too high.

### Assumptions
- The traffic source is the main problem
- Lowering ad cost is the only path
- More spend or more channels will fix growth

### Basic facts
- CAC is shaped by targeting, message fit, trust, conversion, onboarding, retention, and monetization
- Poor positioning makes every channel expensive
- Stronger conversion and retention can justify higher acquisition cost

### Constraint split

Hard constraints:
- Some channels have real pricing floors
- Budget is finite

Soft constraints:
- Current channel mix
- Existing landing-page structure
- Current audience framing
- Internal belief that traffic is the whole problem

### Rebuilt design
Treat CAC as a system metric, not only a traffic metric. Improve:
- audience-message fit
- trust signals
- conversion path
- onboarding speed
- monetization logic

### Minimal test
Run one narrower-positioning test with a clearer promise before expanding channel spend.

# Anti-Patterns

Use this file to avoid misapplying first-principles reasoning.

## Core warning

First-principles reasoning is not the same as:

- being contrarian for style
- dismissing expertise
- rejecting best practices automatically
- ignoring practical constraints
- producing abstract visionary language
- rebuilding everything for no reason

The goal is not rebellion. The goal is accuracy.

## Common failure modes

### 1. Contrarian theater

Bad pattern:

- everyone else is wrong
- ignore the whole industry
- throw out all conventions

Why this fails:
Many conventions solve real problems. The task is to test them, not reject them performatively.

Correct approach:
Keep conventions that survive decomposition. Remove only the ones that do not.

### 2. Empty abstraction

Bad pattern:
Using phrases like:

- rethink from the ground up
- challenge assumptions
- redesign the system
- innovate the workflow

without naming the assumptions, constraints, or changed structure.

Correct approach:
State the assumptions. Name the real constraints. Show the new design. Give the first test.

### 3. Ignoring execution reality

Bad pattern:
Proposing a theoretically elegant solution the user cannot execute because of money, time, team size, regulation, or platform limits.

Correct approach:
Work inside hard constraints. Challenge soft ones. Keep recommendations executable.

### 4. Treating user wording as the real problem

Bad pattern:
The user asks "Should I build an app?" and the answer assumes an app is necessary.

Correct approach:
Translate the request into the real desired outcome first.

### 5. Confusing optimization with redesign

Bad pattern:
Improving each step of a legacy process without checking whether the process should exist in that form at all.

Correct approach:
Ask whether the structure should be reduced, merged, reordered, or replaced before optimizing.

### 6. Throwing away empirical evidence

Bad pattern:
Rejecting known successful patterns because they were not derived from scratch in the current conversation.

Correct approach:
Treat empirical evidence as data. Then test whether it reflects hard reality or historical momentum.

### 7. Overusing iconic examples

Bad pattern:
Forcing every analysis to sound like rockets, Musk, Tesla, or SpaceX.

Correct approach:
Use the structure of the reasoning, not the mythology.

### 8. Ending without a decision

Bad pattern:
The answer explains the world but never tells the user what to do.

Correct approach:
End with a recommendation, priority order, or experiment.

## Self-check before finalizing

Ask:

- Did I identify the real goal?
- Did I list assumptions explicitly?
- Did I confuse a soft constraint with a hard one?
- Did I produce something executable?
- Did I avoid trying to sound profound?

If any answer is no, revise.
